## 1. סקירת מערכת

### 1.1 מטרת המערכת

עמותת "אנו בניך" מבקשת לפתח פלטפורמה דיגיטלית מאובטחת המשמשת כמרחב תמיכה, קהילה, פורום
וייעוץ מקצועי לאוכלוסיות שכול: אלמנים, אלמנות, יתומים ויתומות. המערכת מיועדת לשמש מרחב בטוח,
מוגן ומכובד, תוך שמירה קפדנית על פרטיות המשתמשים, כבודם ורגישותם התרבותית-דתית.

### 1.2 עקרונות יסוד

- **פרטיות כברירת מחדל** — כל מידע אישי ומסמכים רגישים מוצפנים ומוגנים
- **רגישות תרבותית-דתית** — הפרדה ברורה בין מגזרים (חסידי, ליטאי, ספרדי, כללי)
- **הפרדה בין קבוצות** — אלמנים, אלמנות, יתומים ויתומות אינם רואים תוכן שאינו שייך לקבוצתם
- **אימות קפדני** — כניסה למערכת מותנית באישור שני מנהלים ובמסמכים מאמתים
- **הגנה על משתמשים** — מנגנון דיווח, מעקב והסתרה אוטומטית של תוכן בעייתי
- **Audit Trail** — מלא לכל פעולה מנהלתית ורגישה, שקיפות פנימית

### 1.3 קהל יעד

| קבוצה | הגדרה |
|--------|--------|
| אלמנים | מי שאיבד בת זוג |
| אלמנות | מי שאיבדה בן זוג |
| יתומים | מי שאיבד הורה |
| יתומות | מי שאיבדה הורה |

## 2. מבנה משתמשים והרשאות (RBAC)

### 2.1 תפקידים במערכת

#### Admin (מנהל)
- אחריות: ניהול כלל המערכת, אישור הרשמות, ניהול אנשי מקצוע ומבקרים
- ניגש לכל הנתונים ללא הגבלת מגזר/קבוצה (לצורכי ניהול בלבד)
- **לא יכול לקרוא הודעות פרטיות של משתמשים (הגנת פרטיות מוחלטת)**
- יכול להשעות/לבטל חשבונות, לאשר/לדחות הרשמות, לנהל קטלוג אנשי מקצוע
- מקבל התראות על בקשות הרשמה ממתינות וחריגות מיוחדות

#### User (משתמש)
- רואה אך ורק תוכן השייך לקבוצתו (אלמן/אלמנה/יתום/יתומה) ולמגזרו
- יכול לפרסם בפורום הרלוונטי לו, לשלוח הודעות פרטיות לחברי קבוצתו
- יכול לשאול שאלות מקצועיות (ציבוריות או פרטיות) לאנשי מקצוע הרלוונטיים לו
- יכול לדווח על תוכן בעייתי
- לא יכול לראות פרטי משתמשים אחרים (שם בלבד, ללא פרטי קשר)
- לא יכול לגשת לתוכן של קבוצות/מגזרים אחרים

#### Professional (איש מקצוע)
- גישה לפורום מקצועי בלבד (לא לפורום הקהילתי)
- רואה שאלות מופנות אליו/לתחומו, בהתאם למגזרים וקבוצות שאליהן שויך
- יכול לענות על שאלות ציבוריות ופרטיות
- לא רואה מידע מזהה של השואל (לשאלות פרטיות — שם בלבד ברמת כינוי)
- מקבל התראות מייל על שאלות חדשות

#### Moderator (מבקר)
- אחריות: טיפול בדיווחים בתוך הקבוצה/מגזר שהוגדרו לו
- רואה דיווחים ממתינים + ההקשר המלא של ההודעה המדווחת
- יכול: למחוק הודעה, לבטל דיווח, להשעות משתמש (זמנית)
- לא יכול: לראות הודעות פרטיות, לשנות נתוני משתמש, לאשר הרשמות
- מקבל התראות מייל על דיווחים חדשים וחריגות

### 2.2 טבלת הרשאות מפורטת

## 3. חלוקה לקבוצות ומגזרים — מנגנון סינון

### 3.1 מטריצת קבוצות-מגזרים

המערכת פועלת לפי שני צירים עצמאיים. כל משתמש משויך לתא **אחד בלבד** במטריצה:

| קבוצה \ מגזר | חסידי | ליטאי | ספרדי | כללי |
|--------------|-------|-------|-------|------|
| אלמנים | תא 1.א | תא 1.ב | תא 1.ג | תא 1.ד |
| אלמנות | תא 2.א | תא 2.ב | תא 2.ג | תא 2.ד |
| יתומים | תא 3.א | תא 3.ב | תא 3.ג | תא 3.ד |
| יתומות | תא 4.א | תא 4.ב | תא 4.ג | תא 4.ד |

> 16 תאים סה"כ. כל משתמש שייך לתא אחד בלבד.

### 3.2 לוגיקת סינון תוכן

> ⚠️ **הערה:** סעיף זה מסומן "לא מדויק" במסמך המקור — נדרש בירור עם כותב האפיון לפני מימוש (ראה §13, שאלה פתוחה #15).

כל שאילתת תוכן (פורום, שאלות, הודעות) מסוננת לפי הכלל הבא:

- **תוכן ברמת "תא"** — נראה אך ורק לחברי אותו תא (קבוצה + מגזר ספציפיים)
- **תוכן ברמת "קבוצה"** — נראה לכל חברי הקבוצה ללא תלות במגזר
- **תוכן ברמת "מגזר"** — נראה לכל חברי המגזר ללא תלות בקבוצה
- **תוכן "כולם"** — נראה לכל משתמשי המערכת

**מנגנון מימוש:** כל שורת תוכן כוללת שדות `group_visibility` ו-`sector_visibility`. ה-API מחיל WHERE clause אוטומטי לפי פרופיל המשתמש המחובר. לא ניתן לעקוף סינון זה דרך ה-API.

```python
# דוגמה לסינון ב-forum_service.py
posts = db.query(ForumPost).filter(
    or_(
        ForumPost.group_visibility == GroupVisibility.ALL,
        ForumPost.group_visibility == current_user.user_type,
    ),
    or_(
        ForumPost.sector_visibility == SectorVisibility.ALL,
        ForumPost.sector_visibility == current_user.sector,
    )
).all()
```

### 3.3 גישת מבקרים

מבקר משויך לרשימה של תאים (קבוצה+מגזר). הוא רואה דיווחים הנוגעים לתכנים בתאים שהוגדרו לאחריותו בלבד. מבקר יכול לנהל מספר תאים, אך לא כל המערכת (זאת שמורה למנהל).

## 4. מודול תקשורת ופורום

### 4.1 סוגי הודעות

| סוג | יעד | מי יכול לשלוח | מי רואה |
|-----|-----|--------------|---------|
| הודעת תא | קבוצה + מגזר | חברי התא | חברי התא בלבד |
| הודעת קבוצה | קבוצה (כל מגזרים) | כל חברי הקבוצה | כל חברי הקבוצה |
| הודעת מגזר | מגזר (כל קבוצות) | חברי המגזר | חברי המגזר |
| הודעה כוללת | כלל המשתמשים | מנהל בלבד | כל המשתמשים |
| הודעה פרטית | משתמש ספציפי | כל משתמש פעיל | שולח + מקבל בלבד |

### 4.2 תרשים זרימה — פרסום בפורום

1. משתמש לוחץ "הודעה חדשה" בממשק הפורום שלו
2. בוחר סוג הפצה: תא שלי / כל הקבוצה / כל המגזר
3. כותב כותרת ותוכן (עד 5,000 תווים), מצרף קובץ אופציונלי (תמונה, PDF — עד 5MB)
4. מערכת בודקת: משתמש פעיל? ✓ | לא מושעה? ✓ | אין תוכן חסום? (סריקת מילות מפתח בסיסית)
5. אם עבר: ההודעה מפורסמת, מוצפנת, עם timestamp
6. אם נכשל: הודעת שגיאה מתאימה
7. כפתור דיווח זמין מיד עם הפרסום

### 4.3 הודעות פרטיות

- משתמש מחפש נמען לפי שם בלבד (בתוך קבוצתו/מגזרו — **אין חיפוש חוצה-קבוצות**)
- הודעות מוצפנות end-to-end, לא נגישות למנהל ולמבקר
- כפתור דיווח זמין גם בהודעות פרטיות
- אחסון מוגבל: עד 1,000 הודעות לשיחה, ניקוי אוטומטי לאחר 3 שנים
- **מנמ"ש (מנגנון נגד מטרד):** לאחר 3 דיווחים מאותו שולח → חסימה זמנית אוטומטית

## 5. מודול ייעוץ מקצועי

### 5.1 קטלוג אנשי מקצוע

כל איש מקצוע מקוטלג לפי:
- **תחום:** עו"ד, רואה חשבון, פסיכולוג, יועץ כלכלי, רב/דיין, רפואה, סוציאל וורקר, אחר
- **מגזרים:** חסידי / ליטאי / ספרדי / כללי (ניתן לסמן "הכל")
- **קבוצות:** אלמנים / אלמנות / יתומים / יתומות (ניתן לסמן "הכל")

דף פרופיל של איש מקצוע (גלוי למשתמש המורשה) כולל: שם, תחום, תיאור קצר, מגזרים ותחומי עיסוק. **אינו כולל** מייל, טלפון, פרטים מזהים (הגנת פרטיות דו-כיוונית).

### 5.2 תרשים זרימה — שאלה מקצועית

1. משתמש פותח "ייעוץ מקצועי" → רואה רשימת אנשי מקצוע הרלוונטיים לו
2. בוחר: שאלה לאיש מקצוע ספציפי / שאלה כללית לתחום
3. מגדיר: ציבורית (גלויה לכל חברי קבוצתו/מגזרו) / פרטית (גלויה לאיש המקצוע בלבד)
4. כותב את השאלה (עד 2,000 תווים)
5. המערכת שולחת התראת מייל:
   - שאלה לספציפי → מייל ישיר לאיש המקצוע הנבחר
   - שאלה כללית לתחום → מייל לכל אנשי המקצוע בתחום הרלוונטי עם ציון: "שאלה כללית — ייתכן שכבר נענתה"
6. איש המקצוע נכנס, רואה השאלות הממתינות לו, מגיב
7. עם קבלת תשובה — משתמש מקבל התראה (מייל / push notification)

### 5.3 ניהול תגובות ציבוריות

שאלות ציבוריות שנענו הופכות ל"ידע קהילתי" — גלויות לכל חברי אותה קבוצה/מגזר. ניתן לסמן תשובה כ"מועדפת" ע"י שואל, ואנשי מקצוע יכולים להוסיף עדכון/הבהרה לתשובות קיימות.

### 5.4 הגנות על פרטיות בייעוץ

- שאלות פרטיות: גלויות לאיש המקצוע ולשואל בלבד
- שם השואל: מוצג לאיש המקצוע ברמת כינוי בלבד ("אלמנה — ספרדי") אלא אם השואל בחר לחשוף שמו
- מניעת ייצוא: לא ניתן לייצא/להוריד שאלות פרטיות

## 6. מודול דיווח והגנה על משתמשים

### 6.1 תרשים זרימה — מנגנון הדיווח

- כל הודעה (פורום / שאלה / הודעה פרטית) נושאת כפתור "דיווח"
- משתמש לוחץ דיווח → בוחר סיבה מרשימה (הטרדה / תוכן פוגעני / ספאם / אחר)
- הדיווח נרשם: מזהה מדווח, מזהה מדווח-עליו, הודעה מזהה, סיבה, timestamp
- מערכת בודקת: האם זה הדיווח הראשון על הודעה זו?
  - **דיווח ראשון** → התראת מייל למבקר האחראי על אותה קבוצה+מגזר
  - **דיווח שני (ממשתמש שונה)** → הסתרה אוטומטית של ההודעה + התראת דחיפות למבקר
  - **דיווח מ-3+ משתמשים** → ניסיון יצירת קשר נוסף עם מבקר (מייל חוזר)
- מבקר מקבל התראה עם: ההקשר המלא, מספר הדיווחים, זהות המדווחים
- מבקר מחליט:
  - "מחיקת ההודעה" → ההודעה נמחקת, המדווח-עליו מקבל הודעת מערכת, הדיווח מסומן "מוצדק"
  - "ביטול הדיווח" → ההודעה מוצגת מחדש, הדיווח מסומן "שגוי"

### 6.2 מנגנון מעקב וחריגות

| פעולה | סף | חריגה |
|--------|-----|--------|
| מדווח-שגוי תכוף | 5+ דיווחי-שגוי ב-30 יום | התראה למבקר + מגבלת דיווח (3 ליום) |
| מדווח-עליו תכוף מוצדק | 3+ דיווחים-מוצדקים ב-30 יום | התראה למנהל + עיון בהשעיה |
| הפרה חמורה | 2+ הפרות-מוצדקות ב-7 ימים | השעיה אוטומטית זמנית (48 שעות) + התראה למנהל |

### 6.3 לוח בקרת מבקר

- רשימת דיווחים ממתינים (מוקדמים לפי מספר דיווחים)
- היסטוריית דיווחים (מטופלים / מבוטלים)
- כרטיס משתמש: מספר פעמים שדווח עליו, מספר פעמים שדיווח שגוי
- אפשרות השעיה ידנית + הערת מבקר (לתיעוד)

## 7. תהליך הרשמה ואימות

### 7.1 שלבי הגשת הבקשה

1. **פרטים אישיים:** שם מלא, ת"ז/מספר דרכון, תאריך לידה, מגדר, מגזר, סוג (אלמן/ה/יתום/ה)
2. **פרטי קשר:** מייל, טלפון (יאומתו ב-OTP)
3. **העלאת מסמכים:** תעודת פטירה של בן/בת הזוג (אלמן/ה) או ההורה (יתום/ה), תמונת פנים (selfie), תעודת זהות/דרכון
4. **הצהרות:** אישור תנאי שימוש, הסכמה לעיבוד מידע, הצהרה על אמיתות המסמכים
5. **שלב המתנה:** הבקשה עוברת לתור הממתינים לאישור

### 7.2 תהליך האישור הכפול

#### מודל Parallel (מומלץ)
שני מנהלים מקבלים התראה בו-זמנית. כל אחד בוחן ומאשר/דוחה באופן עצמאי. הרשמה מאושרת רק כאשר שני המנהלים אישרו.

**SLA עם Parallel:**
- אם לאחר 72 שעות רק מנהל אחד אישר → תזכורת למנהל השני
- לאחר 7 ימים ללא פעולה → המנהל הבכיר מקבל התראה

#### מודל Sequential (חלופה)
מנהל א׳ בוחן ומאשר ראשון. רק לאחר אישורו, מנהל ב׳ מקבל התראה. יתרון: בקרה עדיפה. חסרון: זמן ארוך יותר.

### 7.3 מצבי חשבון ותרשים זרימה

| סטטוס | משמעות | פעולות אפשריות |
|--------|---------|----------------|
| `PENDING_OTP` | ממתין לאימות מייל/טלפון | שליחת OTP מחדש |
| `PENDING_APPROVAL` | הוגשה, ממתין לשני מנהלים | מנהל: אישור / דחייה / בקשת מידע נוסף |
| `PARTIAL_APPROVED` | מנהל אחד אישר | ממתין למנהל שני |
| `ACTIVE` | שני מנהלים אישרו | גישה מלאה |
| `REJECTED` | אחד/שניים דחו | הודעה + אפשרות ערעור (30 יום) |
| `SUSPENDED` | מנהל/מבקר השעה | השעיה זמנית עם תאריך סיום |
| `CANCELLED` | המשתמש ביקש מחיקה | מחיקת נתונים לפי GDPR |

```
PENDING_OTP ──[OTP אומת]──▶ PENDING_APPROVAL
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              [מנהל א' אישר]  [מנהל דחה]   [בקשת מידע]
                    │               │               │
                    ▼               ▼               ▼
            PARTIAL_APPROVED     REJECTED      (נשאר PENDING)
                    │            + מייל
            [מנהל ב' אישר]     + 30 יום
                    │            לערעור
                    ▼
                 ACTIVE
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
     SUSPENDED             CANCELLED
   (זמן מוגדר)           (מחיקת נתונים)
```

### 7.4 כניסה חוזרת למערכת

לאחר הרשמה מלאה ואישור, המשתמש יכול להיכנס בשתי דרכים:

- **כניסה עם מייל + סיסמה** (ברירת מחדל)
- **כניסה עם Google (Firebase Authentication)** — לא עוקפת הרשמה; זמינה רק למשתמשים שעברו את כל שלבי הרשמה ואושרו על ידי שני מנהלים. (⚠️ ראה §13 שאלה #16 — אופן הקישור בין Firebase UID לחשבון קיים)

## 8. מודל נתונים (ERD)

### 8.1 ישויות ראשיות

#### User (משתמש)
שדות: מזהה ייחודי (UUID), שם פרטי, שם משפחה, ת"ז/מספר דרכון (מוצפן), כתובת מייל (מוצפנת), מספר טלפון (מוצפן), תמונת פנים (URL מאובטח, מוצפן), סוג משתמש [אלמן|אלמנה|יתום|יתומה], מגזר [חסידי|ליטאי|ספרדי|כללי], סטטוס חשבון [ממתין\_לאישור|פעיל|מושעה|מבוטל], תאריך לידה, תאריך יצירה, תאריך אישור, מזהה מאשר ראשון, מזהה מאשר שני.

#### Admin (מנהל)
שדות: מזהה, שם, אחריות מגזרים (רשימה), אחריות קבוצות (רשימה), הרשאות מיוחדות, תאריך מינוי.

#### Moderator (מבקר)
שדות: מזהה, שם, מגזרים אחראיים (רשימה), קבוצות אחראיות (רשימה), מייל התראות.

#### Professional (איש מקצוע)
שדות: מזהה, שם, תחום מקצועי [עו"ד|רואה חשבון|פסיכולוג|יועץ כלכלי|רב/דיין|רפואה|סוציאל וורקר|אחר], מגזרים משויכים (רשימה, יכול להיות "הכל"), קבוצות משויכות (רשימה, יכול להיות "הכל"), מייל, טלפון, תיאור קצר, פעיל (boolean).

#### ForumPost (הודעת פורום)
שדות: מזהה, מזהה יוצר, סוג הפצה [קבוצה|מגזר|קבוצה+מגזר|כולם], קבוצת יעד, מגזר יעד, כותרת, תוכן (מוצפן), תאריך יצירה, תאריך עריכה, מספר דיווחים, סטטוס [גלוי|מוסתר|מחוק].

#### DirectMessage (הודעה פרטית)
שדות: מזהה, מזהה שולח, מזהה מקבל, תוכן (מוצפן), תאריך, קרוא (boolean).

#### ProfessionalQuery (שאלה מקצועית)
שדות: מזהה, מזהה שואל, מזהה איש מקצוע (ספציפי — אופציונלי), תחום (לשאלה כללית), תוכן (מוצפן), ציבורי (boolean), תשובה (מוצפן, אופציונלי), תאריך שאלה, תאריך תשובה, סטטוס [פתוח|נענה|סגור].

#### Report (דיווח)
שדות: מזהה, מזהה מדווח, מזהה מדווח-עליו, סוג יעד [הודעת\_פורום|הודעה\_פרטית|שאלה\_מקצועית], מזהה יעד, סיבה (טקסט חופשי), תאריך, החלטה [ממתין|שגוי|מוצדק], מזהה מבקר מטפל, תאריך החלטה.

#### Document (מסמך)
שדות: מזהה, מזהה משתמש, סוג [תעודת\_פטירה|תמונת\_פנים|ת"ז|דרכון], URL מוצפן, hash, תאריך העלאה, תוקף (אופציונלי).

#### AuditLog (לוג פעולות)
שדות: מזהה, מזהה משתמש מבצע, סוג פעולה, ישות מושפעת, מזהה ישות, IP מקור (מוצפן), timestamp, פרטים (JSON).

## 9. דרישות טכניות ואבטחה

### 9.1 הצפנה

- **מסמכים ותמונות:** AES-256 at rest, אחסון ב-S3/Blob Storage עם presigned URLs בתוקף מוגבל (15 דקות)
- **ת"ז ופרטים מזהים:** מוצפנים ב-DB ברמת שדה (field-level encryption) עם מפתח נפרד לכל משתמש
- **הודעות פרטיות:** מוצפנות בשרת (server-side encryption), עם אפשרות שדרוג ל-E2E בגרסה 2
- **תעבורה:** TLS 1.3 בלבד, HTTPS, HSTS חובה
- **סיסמאות:** bcrypt עם cost factor ≥ 12

### 9.2 ניהול הרשאות (RBAC)

- כל endpoint מוגן ברמת middleware עם בדיקת תפקיד + שייכות לקבוצה/מגזר
- JWT tokens עם תוקף קצר (15 דקות) + refresh token (7 ימים)
- ביטול session מיידי עם השעיה/ביטול חשבון
- עקרון הרשאות מינימליות (Least Privilege) — כל תפקיד מקבל רק מה שנדרש

### 9.3 Audit Trail

- כל פעולה מנהלתית/רגישה נרשמת: מי, מה, מתי, IP
- לוגים אינם ניתנים לעריכה (append-only, חתומים)
- שמירת לוגים: 7 שנים (דרישה ישראלית)
- גישה ל-Audit Log: מנהל בלבד

### 9.4 מדיניות שמירה ומחיקה

| סוג נתון | תקופת שמירה | עם מחיקת חשבון |
|----------|------------|----------------|
| הודעות פורום | 5 שנים | אנונימיזציה (לא מחיקה) |
| הודעות פרטיות | 3 שנים | מחיקה מלאה |
| שאלות מקצועיות ציבוריות | 7 שנים | אנונימיזציה |
| שאלות מקצועיות פרטיות | 5 שנים | מחיקה מלאה |
| מסמכי הרשמה | 10 שנים | מחיקה עם אישור מנהל |
| Audit Logs | 7 שנים | לא נמחקים |
| דיווחים | 5 שנים | אנונימיזציה |

### 9.5 עמידה ב-GDPR ובחוק הגנת הפרטיות הישראלי

- **זכות גישה:** משתמש יכול לבקש העתק של כל המידע שנשמר עליו
- **זכות מחיקה ("זכות להישכח"):** מחיקה תוך 30 יום מבקשה, למעט נתונים בעלי חובה חוקית
- **הסכמה מפורשת:** נדרשת בנפרד לכל מטרת עיבוד
- **DPA:** מינוי ממונה הגנת מידע (Data Protection Officer) — מומלץ
- **Data Breach:** נוהל גילוי פרצה תוך 72 שעות לרשות הגנת הפרטיות

## 10. תשתית וענן

### 10.1 ארכיטקטורת הענן (מוסכם)

| שכבה | שירות | הערות |
|------|--------|-------|
| **Frontend** | Vercel | Angular build, auto-deploy מ-GitHub |
| **Backend** | Railway (מומלץ) | FastAPI + Docker, לא מושבת בחוסר תנועה |
| **Database** | PostgreSQL | Railway managed / Supabase |
| **File Storage** | Cloudflare R2 / AWS S3 | תואם S3-API, presigned URLs |
| **Email** | Resend / SMTP | לשליחת OTP ומיילי מערכת |

> **הערה:** "ענן Vercel" = פרונטאנד על Vercel + בקאנד על שירות נפרד. Vercel אינו מריץ FastAPI ישירות.

### 10.2 אימות עם Google (Firebase Authentication) — נוסף

**דרישה:** משתמשים שכבר עברו הרשמה מלאה ואושרו יוכלו להיכנס בכניסות חוזרות דרך Google, ללא צורך בסיסמה.

**זרימה:**
- הרשמה ראשונה: תמיד דרך מייל + מסמכים + אישור מנהלים (לא ניתן לעקוף)
- לאחר אישור: משתמש יכול לקשר חשבון Google לחשבון הקיים
- כניסה חוזרת: Google OAuth → Firebase → בקאנד מאמת Firebase UID → מחזיר JWT
- הגנה: Firebase UID מקושר ל-User record בלבד אם המשתמש `ACTIVE`

> ⚠️ **TBD (§13, שאלה #16):** מנגנון הקישור הטכני בין Firebase UID לחשבון קיים — יש להגדיר לפני מימוש.

### 10.3 CI/CD

- **GitHub Actions CI:** lint + tests על כל PR (חובה לפני merge)
- **GitHub Actions CD:** deploy אוטומטי ל-Vercel (frontend) + Railway (backend) עם merge ל-main

**סטטוס:** ⏳ בבירור — דרישות לא סגורות

## 12. Use Cases מרכזיים

### UC-01: הרשמת משתמש חדש
- **שחקן:** אלמנה ספרדייה
- **זרימה:** מילוי טופס → OTP → העלאת מסמכים → אימות → ממתין לאישור
- **תוצאה:** חשבון פעיל לאחר אישור שני מנהלים
- **חריגה:** מסמכים לא תקינים → הודעת דחייה + הסבר

### UC-02: פרסום בפורום הקהילתי
- **שחקן:** אלמן חסידי
- **זרימה:** כניסה → פורום → "הודעה חדשה" → בחירת הפצה (תא/קבוצה) → כתיבה → שליחה
- **תוצאה:** הודעה מפורסמת לחברי הקבוצה/תא הנבחרים
- **חריגה:** תוכן חסום → הודעת שגיאה ואפשרות לנסח מחדש

### UC-03: שאלה מקצועית פרטית
- **שחקן:** יתומה ליטאית
- **זרימה:** ייעוץ → בחירת עו"ד ספציפי → שאלה פרטית → שליחה
- **תוצאה:** העו"ד מקבל מייל, עונה במערכת, היתומה מקבלת התראה
- **פרטיות:** שם היתומה אינו מוצג לעו"ד ללא הסכמתה

### UC-04: דיווח על הטרדה
- **שחקן:** אלמנה חסידית
- **זרימה:** פתיחת הודעה בעייתית → "דיווח" → בחירת סיבה
- **תוצאה (דיווח ראשון):** התראה למבקר
- **תוצאה (דיווח שני):** הסתרה אוטומטית + התראת דחיפות

### UC-05: מבקר מטפל בדיווח
- **שחקן:** מבקר אחראי על תא "אלמנות-חסידי"
- **זרימה:** קבלת מייל → כניסה ללוח הבקרה → עיון בהקשר → החלטה
- **תוצאה א:** מחיקת ההודעה + הודעה לשולח
- **תוצאה ב:** ביטול הדיווח + השבת ההודעה

### UC-06: אישור הרשמה ע"י מנהל
- **שחקן:** מנהל א׳
- **זרימה:** קבלת התראה → עיון במסמכים → בדיקת תמונה vs. ת"ז → אישור/דחייה
- **תוצאה:** סטטוס עודכן, מנהל ב׳ מקבל התראה (parallel) או הבקשה מאושרת (sequential)

### UC-07: שאלה מקצועית כללית לתחום
- **שחקן:** אלמן כללי
- **זרימה:** ייעוץ → "שאלה כללית לרואי חשבון" → כתיבה → שליחה
- **תוצאה:** כל רואי החשבון הרלוונטיים מקבלים מייל עם ציון שמדובר בשאלה כללית

### UC-08: מחיקת חשבון ע"י משתמש
- **שחקן:** יתום ספרדי
- **זרימה:** הגדרות → "מחיקת חשבון" → אישור → OTP → בקשה לאדמין
- **תוצאה:** נתונים אישיים נמחקים, הודעות פורום מאונונימות, Audit Log נשמר

### UC-09: מנהל מזהה חריגה
- **שחקן:** מנהל
- **זרימה:** קבלת התראה "3 הפרות מוצדקות ב-7 ימים" → עיון בתיק המשתמש → השעיה
- **תוצאה:** המשתמש מושעה 48 שעות, מקבל מייל הסבר, מנהל רשם הסבר ב-Audit Log

### UC-10: צפייה בשאלה ציבורית שנענתה
- **שחקן:** אלמנה ליטאית
- **זרימה:** ייעוץ → "שאלות ותשובות" → סינון לפי תחום → קריאת שאלה ותשובה
- **תוצאה:** צבירת ידע קהילתי מבלי לחשוף פרטים אישיים

### UC-11: עדכון מנגנון פרופיל איש מקצוע
- **שחקן:** מנהל
- **זרימה:** ניהול → אנשי מקצוע → עריכה → הוספת מגזר/שינוי תחום
- **תוצאה:** עדכון מיידי בקטלוג, משתמשים רלוונטיים רואים את העדכון

### UC-12: התראה על מדווח-שגוי תכוף
- **שחקן:** מבקר
- **זרימה:** מקבל התראה "משתמש X דיווח שגוי 5 פעמים ב-30 ימים"
- **תוצאה:** מבקר בוחן, מוסיף הגבלת-דיווח, תיעוד ב-Audit Log

## 14. מונחון

| מונח | הגדרה |
|------|-------|
| RBAC | Role-Based Access Control — מנגנון הרשאות המבוסס על תפקיד המשתמש |
| Audit Trail | יומן פעולות בלתי ניתן לשינוי המתעד כל פעולה רגישה במערכת |
| OTP | One Time Password — קוד חד-פעמי לאימות זהות |
| AES-256 | תקן הצפנה מתקדם בגודל מפתח 256 ביט, מוכר לשמירת מידע רגיש |
| GDPR | General Data Protection Regulation — תקנות הגנת מידע אירופאיות |
| JWT | JSON Web Token — אסימון אימות דיגיטלי חתום |
| E2E | End-to-End Encryption — הצפנה כך שרק השולח והמקבל יכולים לקרוא |
| ERD | Entity Relationship Diagram — מפת קשרים בין ישויות המידע |
| Presigned URL | קישור זמני ומוגבל-זמן לגישה לקובץ מאובטח באחסון ענן |
| SLA | Service Level Agreement — הסכם רמת שירות ומדדי זמן תגובה |
| מנמ"ש | מנגנון נגד מטרד — חסימה אוטומטית אחרי 3 דיווחים מאותו שולח |
| תא | יחידת הפרדה בסיסית = קבוצה + מגזר (לדוגמה: אלמנות-חסידי) |
